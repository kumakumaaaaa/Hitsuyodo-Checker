## 1. システム概要

### 1.1 目的

日本の病院が厚生労働省に提出するDPCデータ（HファイルおよびEFファイル）を取り込み、看護必要度の集計・分析を行うことで、**入院基本料および各種加算の施設基準（看護必要度割合の閾値）を満たしているかを確認するためのツール**。

手作業による集計・判定業務を自動化し、要件充足状況を視覚的に把握できるようにすることで、適切な診療報酬算定を支援する。

### 1.2 対象ユーザー

- 医療事務担当者（診療報酬算定業務）
- 看護管理者（病棟単位の看護必要度管理・モニタリング）

### 1.3 入力データ

| ファイル | 内容 | 主な用途 |
|---------|------|---------|
| **Hファイル** | 患者属性・病棟コード・入退院情報 | 病棟コードの抽出、患者在院期間の特定 |
| **EFファイル** | 診療行為明細（レセプト電算コード含む） | **A項目・B項目のスコア参照（本システムの主目的）**、C項目（手術等）の判定、看護必要度割合の算出 |

> **前提:** 1レコード＝HファイルとEFファイルの1セット＋初期設定情報。  
> 対象期間は原則**3ヶ月分**を想定するが、任意期間の指定にも対応する。

### 1.4 技術スタック

| レイヤー | 技術 |
|---------|------|
| フロントエンド | React / Next.js / TypeScript |
| データベース | PGLite（ブラウザ内軽量DB）+ Zustand（オンメモリ状態管理） |
| スタイリング | Tailwind CSS |
| 状態管理 | Zustand |

## 2. 画面構成と機能要件

### 2.1 画面一覧（概要）

```
共通ヘッダー（全画面）
  ├─ 左：ロゴ（クリックでホームへ）
  ├─ 右：歯車アイコン → 設定ポップアップ（オーバーレイ）
  └─ 右端：アカウントアイコン（頭文字）→ ホバーでアカウント情報・ログアウト

設定ポップアップ
  └─ 病棟コード・入院料デフォルト設定

ホーム画面（レコード一覧）
  └─ 新規レコード作成フロー
       ├─ Step 1: 基本設定（評価方式・タイトル・対象期間・ファイルアップロード）
       └─ Step 2: 病棟設定・確認・レコード生成
  └─ レコード詳細画面（タブ切替）
       ├─ Tab 1: ファイル取り込み状況（初期設定・入院料の確認）
       ├─ Tab 2: 基準割合（重症該当患者割合の表・グラフ）
       ├─ Tab 3: 看護必要度詳細（日次患者レコードのテーブル一覧・Excel出力） ✅
       ├─ Tab 4: 分析（A・B・C項目の病棟別・日数別変化）
       └─ Tab 5: 看護必要度Ⅰ・Ⅱ比較（看護必要度Ⅰ選択時のみ有効）
```

---

### 2.2 共通ヘッダー

**目的:** 全画面で共通して表示されるナビゲーション領域。

**機能要件:**

- 左端にロゴ（「看護必要度アナリシスソフト（仮）」）を配置し、クリックでホーム画面に遷移する
- 右側に歯車アイコンを配置し、クリックで設定ポップアップをオーバーレイ表示する
- 右端にアカウントアイコン（ログイン中アカウントの頭文字）を配置し、ホバーでアカウント情報とログアウトボタンを表示する

### 2.3 設定ポップアップ

**目的:** システム全体のデフォルト設定とマスタ編集をまとめて行う画面。ヘッダーの歯車アイコンからオーバーレイで呼び出す。

**機能要件:**

- **病棟コード・入院料デフォルト設定:** 病棟コードごとにデフォルトの病棟名称・入院料を設定・保存できる。新規レコード作成時、ファイルから検出された病棟コードにこのデフォルト値が自動適用される
  - 病棟コードの重複登録は不可（同一コードは上書き更新）
  - 入院料はカテゴリ別（一般病棟・地域包括ケア病棟等）のドロップダウンで選択
  - 選択した入院料の判定パターン（P1/P2/P3等）をプレビュー表示
  - データはlocalStorageに永続化

### 2.4 ホーム画面（レコード一覧）

**目的:** 作成済みレコードの一覧管理と新規作成の起点となる画面。NotebookLMライクなテーブル形式のUIを採用する。

**機能要件:**

- レコードをテーブル形式で一覧表示する。列構成は以下の通り
  - タイトル（自動生成をデフォルトとし、ユーザーが編集可能。例：「2025年10月分」）
  - 対象期間（例：2025/08〜2025/10）
  - 病棟数（含まれる病棟コード数）
  - 作成日
  - 充足状況サマリー（充足 / 要確認 / 未充足 のバッジ表示）
- 右上に目立つ「＋ 新規作成」ボタンを配置する
- レコードの行をクリックすると詳細画面に遷移する
- レコードの削除機能を持つ（削除後は一覧から除外、復元不可）

### 2.5 新規レコード作成フロー

#### Step 1: 基本設定・ファイルアップロード

**目的:** 評価方式の選択、レコードの基本情報設定、HファイルとEFファイルのアップロードを行う。

**機能要件:**

- **評価方式の選択**
  - 看護必要度Ⅰ または 看護必要度Ⅱ をラジオボタンで選択する
  - ⅠとⅡでは判定基準（評価方式・閾値）が異なるため必須設定
- **レコードタイトルの設定**
  - 自動生成をデフォルトとし、ユーザーが編集可能（例：「2025年10月分」）
- **対象期間の設定**
  - 分析対象とする期間を開始年月・終了年月で指定する
- **ファイルアップロード**
  - HファイルおよびEFファイルをそれぞれアップロードできる（ドラッグ＆ドロップ対応）
  - アップロード後、ファイルの基本情報（ファイル名・サイズ・行数）をプレビュー表示する
    - ファイル形式チェック：`.txt`, `.csv`, `.tsv` のみ許可。それ以外のファイルはエラー表示する
    - ファイルサイズチェック：100MB を超えるファイルはエラー表示する
    - 空ファイルチェック：0バイトのファイル、またはデータ行が0行のファイルはエラー表示する
    - ヘッダ形式チェック：ファイルアップロード時に先頭行を読み込み、ファイル種別（H/EF）としての妥当性を検証する。要件を満たさない場合はそれぞれ「H/EFファイルではありません」とエラー表示する
      - Hファイル：1列目が「施設コード」、列数が8列以上、2列目「病棟コード」、3列目「データ識別番号」であること
      - EFファイル：1列目が「施設コード」、列数が30列以上、2列目「データ識別番号」、3列目が「退院年月日」から始まること（「退院年月日（西暦）」等の表記揺れを許容）
    - 対象期間チェック：開始月≦終了月であること。また、最大3ヶ月以内であること。エラー時はメッセージを表示する
    - 終了日補正：終了月（periodTo）は、データを保存する際にその月の「月末日」に自動補正する
    - データ期間合致チェック：Hファイルの実施年月日（Hファイル）、EFファイルの実施年月日（EF-24）を読み取り、それぞれのデータの最小日および最大日が、指定した「対象期間」の範囲内に含まれているかを検証する。一部でも対象期間外のデータがある場合、または対象期間のデータが全く含まれない場合はエラーとして処理を中断し、対象となるファイル（H/EF）と対象期間の不一致をメッセージで表示する
    - 評価方式整合性チェック：Hファイル内のテキストを解析し「ASS0013」または「ASS0062」が含まれているか検証する。看護必要度Ⅰ選択時に含まれていない場合は「このファイルは看護必要度Ⅰではありません」エラーを表示。看護必要度Ⅱ選択時に含まれている場合は「このファイルは看護必要度Ⅰです」エラーを表示する。
    - エラー発生時はドロップゾーンを赤枠で強調し、エラー内容をメッセージで表示する
    - 正しいファイルを再選択（あるいは対象期間を修正）するとエラーがクリアされる

#### Step 2: 病棟設定・確認・レコード生成

**目的:** ファイル取込後に病棟コードを設定し、設定内容を確認してレコードを確定する。

**機能要件:**

- **Step 1 のサマリー表示**
  - Step 1 で入力した基本情報（タイトル・評価方式・対象期間・ファイル）を確認用に表示する
- **病棟コード紐付け設定**
  - H/EFファイルから病棟コードを自動抽出し、固定表示（編集不可）する
  - Shift-JISエンコーディングに対応し、不正な病棟コードは自動フィルタする
  - 病棟名称は必須入力とする（未入力時は赤枠バリデーション、生成ボタン無効化）
  - 入院料をカテゴリ別ドロップダウンから選択できる（一般病棟 / 地域包括ケア病棟）
  - 選択した入院料の判定パターン（P1/P2/P3等）をサブテキストで表示する
  - デフォルト設定が存在する病棟コードには、デフォルト値（名称・入院料）を自動適用する
  - 「この病棟設定をデフォルトとして保存する」チェックボックスで、入力値をデフォルト設定に反映できる（病棟コード重複時は上書き）
- **レコード生成**
  - 確定ボタン押下でデータ処理を実行し、レコードを生成する
  - 処理中は実処理連動のプログレスUI（6ステップ: ①DB保存 → ②ファイルパース → ③B項目計算 → ④A・C項目計算 → ⑤施設基準判定 → ⑥データ生成）を表示する
  - 処理完了後、自動的に生成されたレコードの詳細画面（Tab 1）へ遷移する

### 2.6 レコード詳細画面

レコード詳細はタブ切替形式で5つのサブ画面で構成する。

---

#### Tab 1: ファイル取り込み状況

**目的:** このレコードに設定されたファイルと初期設定内容を確認する。

**機能要件:**

- アップロードされたHファイル・EFファイルのファイル名・行数・データ期間を表示する
- 設定された病棟コードと入院料の紐付け一覧を表示する
- 看護必要度区分（Ⅰ or Ⅱ）・対象期間を表示する
- 設定内容の再編集導線を設ける（編集するとデータを再処理する）

---

#### Tab 2: 基準割合

**目的:** 入院基本料・加算ごとの重症該当患者割合を確認し、施設基準の充足状況を把握する。

**機能要件:**

- 設定された入院基本料ごとに、看護必要度の重症該当患者割合（実績値）と閾値を比較表示する
- 表形式とグラフ形式の両方で表示する
- 充足・未充足・ボーダーライン（閾値±5%以内）を色分けで視覚的に示す
- 入院基本料と加算はそれぞれ独立した基準として表示する（同時充足が必要なため）

---

#### Tab 3: 看護必要度詳細

**目的:** 日次の患者別看護必要度レコードを一覧で確認・出力する。

**機能要件:**

- 患者ごと・日次のA項目・B項目・C項目スコアをテーブル形式で一覧表示する
- 列の並び順: 患者基本情報（病棟コード・病棟名称等）→ 得点・基準（A/B/C得点・基準①②③）→ A/B/C個別項目（詳細）
- カラムヘッダは項目名称のみ表示（記号A1・B2等は使用しない）、ヘッダは固定（sticky）でデータ行のみスクロール
- 基準列には条件式を表示（例: 「基準① A≧３ or C≧１」）
- 病棟名称・データ識別番号での絞り込みができる
- テーブルデータを**Excel形式**でエクスポートできる（テーブルと同じ列構成・表示名）

---

#### Tab 4: 分析

**目的:** A・B・C各項目の傾向を病棟別・時系列で分析する。

**機能要件:**

- A項目・B項目・C項目それぞれのスコア分布・推移をグラフで表示する
- 病棟別の比較表示ができる
- 日次・週次・月次など集計単位を切り替えられる
- 具体的なグラフ種類・レイアウトは画面設計フェーズで決定する（現時点では未定）

---

#### Tab 5: 看護必要度Ⅰ・Ⅱ比較

**目的:** 看護必要度ⅠとⅡの判定結果を比較分析する。

**機能要件:**

- **看護必要度Ⅰが選択されたレコードのみ閲覧可能とする**
- 看護必要度Ⅱが選択されたレコードでは、タブ自体はレイアウト上に表示するが**グレーアウト**状態とし、マウスホバー時に「この画面は看護必要度Ⅰが選択されている場合のみ閲覧できます」というツールチップを表示する
- 比較の具体的な内容・グラフ構成は将来の実装フェーズで決定する（現時点では未実装予定）

## 3. 非機能要件

### 3.1 動作環境

- 対応ブラウザ：Google Chrome（最新版）・Microsoft Edge（最新版）
- 動作方式：クライアントサイド完結（外部サーバー不要）
- データ永続化：病棟設定等の一部設定のみlocalStorageに保存。分析対象データは永続化せずオンメモリで処理する

### 3.2 パフォーマンス

- HF・EFファイルの解析処理：1ヶ月分（数千〜数万行）を30秒以内に完了すること（目標値）
- 集計・グラフ表示：操作後3秒以内に表示すること

### 3.3 保守性・拡張性

- DBアクセスはリポジトリ層に集約し、将来的なサーバーサイドDB（Supabase等）への移行コストを最小化する
- 入院料・加算の判定ルール・閾値はマスタデータで管理し、診療報酬改定への対応をコード変更なしで行えるよう設計する

### 3.4 データ安全性

- データはすべてローカルブラウザ内に保存され、外部サーバーへの送信は行わない（初版）
- ブラウザキャッシュクリアによるデータ消失リスクをユーザーに対して画面上で明示する

## 4. 制約・前提条件

- 本システムはローカル環境での試作（PoC）を目的とし、複数端末でのデータ共有は対象外とする（初版）
- 診療報酬ルールは2024年度改定を基準とし、改定対応はマスタデータの更新で対応する
- ユーザー認証・権限管理は初版スコープ外とする
- HファイルおよびEFファイルのフォーマットはDPC標準仕様に準拠することを前提とする
